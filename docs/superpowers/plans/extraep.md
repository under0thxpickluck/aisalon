# LIFAI GiftEP 完全統合仕様書

## EP送信時全量GiftEP変換モデル・完全版

# 0. 本仕様の前提

本仕様において、GiftEPは「ユーザーが保有するEPを他ユーザーへ送信したときに、その送信量の全量がGiftEPへ自動変換され、受信者へ付与される、LIFAI内限定の消費型クレジット」として扱う。
GiftEPは通貨・資産・換金手段ではなく、LIFAI内の指定機能でのみ利用可能な期限付きクレジットである。
本仕様は、既存のEP/BP構造を壊さずに追加実装すること、UI/UX上は「送金」ではなく「ギフト」「応援」「期限付きクレジット」として体験させることを前提とする。

# 第1部　GiftEP利用規約案

## 第◯条（GiftEPの定義）

1. GiftEPとは、当社が提供するサービス「LIFAI」内において利用可能な、ユーザーが保有するEPを他のユーザーに送信した際に自動的に変換されて付与される消費型ポイントをいいます。
2. ユーザーがEPを他のユーザーへ送信した場合、当該EPは送信時点で全量がGiftEPに変換され、送信先ユーザーに付与されます。
3. GiftEPは、資産、通貨、有価証券その他これらに類する財産的価値を有するものではありません。
4. GiftEPは、当社が指定するサービス内機能（MUSICBOOST、ワークフロー等）においてのみ利用可能とします。
5. GiftEPは、当社が別途定める利用条件、禁止事項、利用範囲および有効期限に従ってのみ利用できるものとします。

---

## 第◯条（GiftEPの性質）

1. GiftEPは換金性を有さず、現金、暗号資産、電子マネーその他の経済的価値と交換することはできません。
2. GiftEPはユーザーの財産権の対象とはならず、当社はその価値を保証しません。
3. GiftEPは有効期限を有し、当社が定める期間（付与後30日または60日）を経過した場合、自動的に失効します。
4. GiftEPはEPとは異なる独立したポイントであり、GiftEPをEPへ戻すこと、GiftEPを通常EPとして扱うこと、GiftEPを別の経済的価値へ変換することはできません。
5. GiftEPはLIFAI外では何らの効力も有さず、外部サービス、第三者プラットフォーム、外部取引、外部決済手段等に利用することはできません。

---

## 第◯条（ギフト機能）

1. ユーザーは、当社が定める条件を満たした場合に限り、自己が保有するEPを他のユーザーに送信することができます。
2. 前項の送信が行われた場合、送信されたEPは自動的にGiftEPに変換され、受信者に付与されます。
3. ギフト機能の利用には、本人確認（KYC）が必要となります。
4. 一度付与されたGiftEPは、いかなる理由があっても第三者への再譲渡（再送）はできません。
5. ユーザーは、EPを送信することにより当該EPを失い、その後返還、払戻し、取消しまたは原状回復を求めることはできません。
6. 当社は、必要に応じて送信上限、月間上限、送信頻度制限、受信制限その他の利用条件を定めることができます。

---

## 第◯条（利用制限）

1. GiftEPは以下の用途にのみ利用可能とします。

   * 当社が提供するデジタルコンテンツの購入
   * 当社が提供する機能の利用
   * 当社が提供する運営公式コンテンツの消費
   * その他当社が明示的にGiftEP利用可と定めたサービス内機能

2. GiftEPは以下の用途には一切使用できません。

   * ユーザー間取引（マーケット、出品、購入等）
   * 換金性を有するサービス
   * 第三者への再配布または譲渡
   * 外部サービス連携
   * 外部の役務、商品、サービスの対価としての利用
   * GiftEPを介した価値移転、報酬支払い、謝礼、返礼その他これに類する行為

3. GiftEPを用いて取得したコンテンツについて、当社は商用利用の制限を設けることがあります。

---

## 第◯条（禁止事項）

ユーザーは以下の行為を行ってはなりません。

1. GiftEPの売買、交換、現金化またはこれらを目的とする行為
2. 第三者（外部業者を含む）に対するGiftEPの譲渡または譲渡の約束
3. GiftEPの価値を対価として、金銭またはこれに類する利益を受け取る行為
4. 当社の許可なくGiftEPの外部流通を促進する行為
5. 不正取得、複数アカウントの利用、その他不正な方法によるGiftEPの利用
6. GiftEPの対価として、以下のいずれかの利益を受け取る行為

   * その他経済的価値を有する一切の利益
7. KYCを回避し、または他人名義等を利用してGiftEPを送受信する行為
8. 外部相場、買取、転売、仲介、交換募集などGiftEPに市場価値があるかのように見せる行為
9. 受信したGiftEPをさらに第三者へ再移転させることを目的とした名義貸し、代理受領その他の迂回行為

# 第2部　GiftEPシステム仕様書

## 1. 概要

GiftEPは、LIFAI内でユーザーが保有するEPを他ユーザーに送信した際、その送信量の全量が自動的に変換されて付与される「消費専用クレジット」である。
本システムは、通貨性・換金性を排除し、あくまでサービス内消費に限定することで、法的リスクを抑えた設計とする。
GiftEPは、通常EPとは明確に切り離され、用途・UI・保存方法・利用ルール・有効期限・不正制御を別管理とする。 

---

## 2. 基本設計

### 2.1 通貨種別

| 種類     | 説明                    |
| ------ | --------------------- |
| BP     | ボーナスポイント（通常利用）        |
| EP     | エクストラポイント（通常利用）       |
| GiftEP | ギフト専用ポイント（EP送信時に自動変換） |

### 2.2 GiftEPの特性

* EP送信時に全量自動変換される
* ユーザー間で受け渡される
* 再送不可（受取後の再ギフト不可）
* 有効期限あり（付与から30〜60日）
* 手数料なし
* KYC必須
* マーケット使用不可
* ユーザー間取引不可
* 運営提供コンテンツのみ使用可能

---

## 3. 利用範囲

### 3.1 使用可能

* MUSICBOOST
* ワークフローコンテンツ
* その他運営提供機能
* 今後追加される運営公式機能で、GiftEP利用可と定義されたもの

### 3.2 使用禁止

* マーケット購入
* マーケット出品
* ユーザー間取引
* 換金性のある機能
* 外部サービス連携
* 外部交換、外部譲渡、外部買取前提の利用

---

## 4. ギフト機能仕様

### 4.1 送信条件

* 本人確認（KYC）済みユーザーのみ
* 送信元はEPを保有している必要がある
* EP送信時に、送信したEPは送信者口座から減算される
* 送信したEPの同量がGiftEPとして受信者に付与される
* 最低送信額：未設定または運営が設定
* 最大送信額：制限あり（例：1回 10,000 EP相当）
* 月間上限：50,000 EP相当
* 自分自身への送信は禁止
* 不正検知対象ユーザーには送信制限を行う場合がある

### 4.2 受信仕様

* 受信後は自動的にGiftEP残高に加算
* 受信したGiftEPは再送不可
* 有効期限が個別に付与される
* GiftEPは受信者の通常EPへ統合されない
* GiftEPは専用残高として管理される

### 4.3 UIフロー

```plaintext
送信画面
↓
送信先選択
↓
金額入力
↓
確認画面（規約同意）
↓
送信確定
↓
GAS処理
↓
EP減算 / GiftEP付与
↓
残高更新
↓
完了表示
```

## 7. GAS Action

| action           | 内容             |
| ---------------- | -------------- |
| gift_send        | ギフト送信処理        |
| gift_balance     | 残高取得           |
| gift_history     | 履歴取得           |
| gift_expire      | 期限切れ処理         |
| gift_use         | 運営機能でのGiftEP消費 |
| gift_rules_check | 利用可否判定         |

### 7.1 actionごとの役割

#### gift_send

* 送信者のKYC確認
* 送信上限チェック
* 送信元EP残高確認
* 送信元EPの減算
* 受信者へのGiftEP加算
* 再送不可制御
* 履歴記録

#### gift_balance

* 利用可能GiftEP
* 期限が近いGiftEP
* 次回失効日
  を返す

#### gift_history

* 贈った履歴
* 受け取った履歴
* 失効履歴
  を返す

#### gift_expire

* 毎日1回実行
* 期限切れGiftEPを users / transactions に反映

#### gift_use

* 運営コンテンツでGiftEPを消費する
* 期限が近い残高から優先消費
* usage_logs に記録

#### gift_rules_check

* 特定機能でGiftEPが使えるか判定
* マーケットやユーザー間取引では false を返す

## 8. 有効期限処理

### 8.1 仕様

* 毎日1回バッチ処理
* 期限切れGiftEPを自動削除
* users.gift_ep_expiry_map を更新
* gift_ep_balance を再計算
* gift_transactions.status を expired に更新

### 8.2 GAS例

```javascript
function expireGiftEP() {
  // expiry_date < today のものを削除
}
```

---

## 9. GiftEP残高・期限ロジック

### 9.1 GiftEP残高の考え方

* `gift_ep_balance` は合計残高
* `gift_ep_expiry_map` は期限別内訳

例：

```plaintext
total: 1500
expiry_map:
2026-05-01: 500
2026-05-15: 1000
```

### 9.2 消費順

GiftEPは**失効日が近い順に自動消費**する。

例：

* 2026-05-01 の500を先に使う
* 足りなければ 2026-05-15 から消費

### 9.3 失効バッチ

* `gift_expire` を time-based trigger で毎日1回実行
* 今日以前の expiry_date を検索
* users.gift_ep_expiry_map から該当額を削除
* gift_ep_balance を再計算
* gift_transactions.status を expired に更新

### 10.2 対応

* 一時停止
* 管理者確認
* 追加KYC
* 送信制限
* GiftEP没収
* アカウント停止またはBAN

# 第5部　LIFAI GiftEP 実装工程書（ClaudeCode向け）

## 1. 目的

GiftEP機能を、既存のLIFAI構造をなるべく壊さずに追加実装する。
GiftEPは以下の性質を持つ。

* EP送信時に自動変換される
* 再送不可
* 手数料なし
* KYC必須
* 有効期限あり（30〜60日）
* マーケット使用不可
* ユーザー間取引不可
* 運営コンテンツのみ使用可（MUSICBOOST / ワークフロー等）

UI/UX上は「送金」ではなく「LIFAI内限定の期限付きギフトクレジット」として扱う。 

---

## 2. 基本方針

### 実装ルール

* 既存のEP/BP構造は壊さない
* GiftEPは新しいポイント種別として追加する
* 既存APIや既存GAS actionの名称変更はしない
* 既存マーケット処理には極力手を入れず、GiftEP側で使用禁止制御を行う
* UIは「ウォレット」「送金」っぽく見せず、「ギフト」「贈る」「使う」に統一する
* まずはバックエンド基盤を作り、その後UIを載せる
* 最後に利用規約同意・警告表示・不正検知を追加する

---

* gift_send
* gift_balance

### 2-2. actionごとの役割

* gift_send：送信者のKYC確認、送信上限チェック、EP減算、受信者へのGiftEP加算、履歴記録

## Phase 4：GiftEP残高・期限ロジック

### 4-1. GiftEP残高の考え方

* `gift_ep_balance` は合計残高
* `gift_ep_expiry_map` は期限別内訳

### 4-2. 消費順

GiftEPは失効日が近い順に自動消費する。

### 4-3. 失効バッチ

`gift_expire` を time-based trigger で毎日1回実行する。

---

## Phase 5：ギフト送信処理

### 5-1. 送信前チェック

gift_send 内で以下を必須チェックする。

* 送信者がKYC済みか
* 送信先が有効なユーザーか
* 送信額が0以下でないか
* 送信額が1回上限以内か
* 月間送信総額が月間上限以内か
* 送信者のEP残高が足りるか
* 送信対象が自分自身でないか
* 不正フラグ中でないか

### 5-2. 再送不可の実装

* GiftEPは通常EPと完全分離
* GiftEPは受信後も GiftEP のまま保持
* 受信したGiftEPは sendable = false とする
* gift_send は GiftEPからの送信を受け付けない
* 送信元は通常EPのみ

# 最終設計判断

GiftEPのUXは、

**“自由なお金”に見せるほど危険**
**“期限付きの応援ギフト”に見せるほど安全**

したがって、機能としては「EP送信→GiftEP変換」であっても、体験としては**LIFAI内限定のプレゼントチケット**として見せるのが最適。これは元文書のUI/UX方針とも一致している。
