## 3. 既知のバグ・技術的負債

### 🔴 クリティカル（未対応）

#### C-4: GAS の SECRET_KEY デフォルト値

```
場所: GAS Code.gs
問題: SECRET_KEY が ScriptProperties に未設定の場合、
      "LIFAITOMAKEMONEY" がデフォルトで使用される
影響: パスワードハッシュが推測可能になる

対処:
  1. GAS エディタ → プロジェクトの設定
  2. スクリプトプロパティ に SECRET_KEY を設定
  3. 値: ランダムな32文字以上の文字列
```

---

### 🟠 高優先度（未対応）

#### H-4: login action が non-approved ステータスを区別できない

```
場所: GAS Code.gs の login アクション
問題: approved 以外は全て { reason: "pending" } を返す
      pending_payment / pending_error / paid を区別できない
影響: ユーザーが支払い済みでも「審査中」と表示される

対処:
  GAS の login アクションで status を判定して返すよう修正:
  - "pending_payment" → { reason: "not_paid" }
  - "paid" → { reason: "awaiting_approval" }
  - "approved" → { ok: true, ... }
```

---

### 🟡 中優先度（未対応）

#### M-3: リセットメール URL ハードコード

```
場所: GAS Code.gs の sendResetMail_()
問題: https://lifai.vercel.app/reset?token=... に固定
影響: ドメイン変更時にメールが壊れる

対処:
  GAS ScriptProperties に SITE_URL を追加:
  PropertiesService.getScriptProperties()
    .getProperty("SITE_URL") + "/reset?token=" + token
```

#### M-4: プラン金額の二重管理

```
場所: GAS planToGrant_() と app/purchase/page.tsx
問題: 同じ金額テーブルが両方に定義されており、
      片方を変更しても自動で同期されない
影響: プラン追加・変更時に不整合が発生するリスク

対処案A: GAS から金額テーブルを返す API を作成
対処案B: Next.js 側を source of truth として GAS に渡す
```

#### M-5: TypeScript strict モード無効

```
場所: tsconfig.json
問題: "strict": false のため型安全性が低下
影響: null/undefined エラーが実行時まで検出されない

対処:
  "strict": true に変更し、型エラーを段階的に修正
```

---

### 🔵 低優先度（未対応）

| ID | 問題 | 推奨対処 |
|---|---|---|
| L-1 | CSRF 対策なし | SameSite Cookie or CSRF トークン実装 |
| L-2 | レート制限なし | `/api/auth/login` に Vercel KV でレート制限 |
| L-3 | IPN ログに機密情報 | ログのサニタイズ（金額・通貨のみ記録） |
| L-4 | GAS に重複関数 | `getSheetValuesSafe_` を削除、`getValuesSafe_` に統一 |
| L-5 | テストメールハードコード | GAS のテスト関数から本番メールアドレスを削除 |
| L-6 | `useMemo` 依存配列 | 不要な再計算を抑制するよう最適化 |
| L-7 | `applyId` 衝突可能性 | UUID v4 に変更（`crypto.randomUUID()`） |

## 6. スケーラビリティ課題

### 現在の制限

| 項目 | 現在 | 限界 | 対処 |
|---|---|---|---|
| データストア | Google Sheets | ~10万行 | DB 移行 |
| ジョブストア | Node.js メモリ | インスタンス共有不可 | Redis |
| メール送信 | GAS (Gmail) | 100通/日 | SendGrid |
| 音楽生成同時実行 | 制限なし | Replicate レート制限 | キュー実装 |
| applyId | Date.now() | 同時生成で衝突 | UUID v4 |
| ref_tree 再生成 | 全件再計算 | 大規模で遅い | 差分更新 |

### GAS パフォーマンス

現在 `getDataRange()` で全件取得を行っているため、ユーザーが数千人を超えると遅延が発生する可能性がある。

**対処案:**
- 検索条件を付けた範囲指定（`getRange()` + 条件マッチ）
- シートをユーザー数ごとに分割
- キャッシュサービス（`CacheService`）の活用
